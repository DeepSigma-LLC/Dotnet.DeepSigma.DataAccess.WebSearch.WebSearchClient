I’d build it as a **typed .NET client library** around SearXNG’s HTTP API, with a small core domain model, strong configuration, and resilience built in.

At a high level, SearXNG exposes `/search` and can return JSON when you pass `format=json`, but the official docs note that JSON/CSV/RSS may be disabled on some public instances, in which case those requests can fail. That makes **instance capability detection** and **graceful failure behavior** important in a production package. ([SearXNG Documentation][1])

## Recommended package shape

I would split it into 4 layers:

1. **Abstractions**

   * `ISearxngClient`
   * request/response models
   * error types

2. **Transport**

   * `HttpClient`-based implementation
   * auth/header support if you run a protected internal instance
   * query building and response parsing

3. **Resilience + observability**

   * retry/timeout/rate-limit handling
   * structured logging
   * metrics/tracing hooks

4. **Dependency injection package**

   * `AddSearxngClient(...)`
   * typed client registration
   * options validation

That shape fits Microsoft’s recommended `IHttpClientFactory` approach for configurable, DI-friendly HTTP clients, and it keeps your core API independent from ASP.NET specifically. ([Microsoft Learn][2])

## Public API design

Keep the surface small and stable.

```csharp
public interface ISearxngClient
{
    Task<SearchResponse> SearchAsync(
        SearchRequest request,
        CancellationToken cancellationToken = default);
}
```

### Request model

Expose only the knobs SearXNG users actually need:

```csharp
public sealed record SearchRequest(
    string Query,
    int? Page = null,
    string? Language = null,
    string? TimeRange = null,
    SafeSearchLevel? SafeSearch = null,
    IReadOnlyList<string>? Categories = null,
    IReadOnlyList<string>? Engines = null);
```

### Response model

Normalize into a provider-neutral shape where possible:

```csharp
public sealed record SearchResponse(
    IReadOnlyList<SearchResult> Results,
    SearchMetadata Metadata,
    IReadOnlyList<SearchWarning> Warnings);

public sealed record SearchResult(
    string Title,
    string Url,
    string? Snippet,
    string? Engine,
    IReadOnlyList<string>? ParsedUrls = null);

public sealed record SearchMetadata(
    string InstanceBaseUrl,
    string Query,
    int? Page,
    TimeSpan Duration,
    bool Partial,
    int ResultCount);
```

This gives you room later to support other backends behind the same interface.

## Configuration design

Use options + validation.

```csharp
public sealed class SearxngOptions
{
    public required Uri BaseUri { get; init; }
    public TimeSpan Timeout { get; init; } = TimeSpan.FromSeconds(10);
    public string SearchPath { get; init; } = "/search";
    public string Format { get; init; } = "json";
    public string? UserAgent { get; init; }
    public bool ProbeInstanceOnStartup { get; init; } = false;
}
```

Important validations:

* `BaseUri` must be absolute
* `Timeout` bounded to something sane
* `SearchPath` should default to `/search`
* `Format` should usually be forced to `json`

SearXNG officially documents `GET /search` and `POST /search`, and the JSON response path is driven by the `format` parameter. ([SearXNG Documentation][1])

## Transport choices

Use a **typed client with `IHttpClientFactory`**.

Microsoft recommends `IHttpClientFactory` for creating and configuring logical HTTP clients, and also recommends the typed-client approach when you want configurability and DI integration. It also helps avoid common lifetime/socket issues associated with creating clients ad hoc. ([Microsoft Learn][2])

### Registration

```csharp
public static class ServiceCollectionExtensions
{
    public static IServiceCollection AddSearxngClient(
        this IServiceCollection services,
        Action<SearxngOptions> configure)
    {
        services.AddOptions<SearxngOptions>()
            .Configure(configure)
            .Validate(o => o.BaseUri is not null && o.BaseUri.IsAbsoluteUri,
                "BaseUri must be absolute");

        services.AddHttpClient<ISearxngClient, SearxngClient>((sp, http) =>
        {
            var options = sp.GetRequiredService<IOptions<SearxngOptions>>().Value;
            http.BaseAddress = options.BaseUri;
            http.Timeout = options.Timeout;

            if (!string.IsNullOrWhiteSpace(options.UserAgent))
                http.DefaultRequestHeaders.UserAgent.ParseAdd(options.UserAgent);

            http.DefaultRequestHeaders.Accept.ParseAdd("application/json");
        });

        return services;
    }
}
```

## Resilience strategy

For production, I would add:

* timeout
* retry for transient failures
* circuit breaker for sustained upstream failures
* optional rate limiter for noisy callers

Current Polly guidance centers on **resilience pipelines**, and Microsoft’s `Microsoft.Extensions.Http.Resilience` package is now the official path for resilient `HttpClient` pipelines in modern .NET. Polly v8 describes retry, timeout, circuit breaker, hedging, fallback, and rate limiter as built-in resilience strategies. ([Polly Docs][3])

### Good defaults

* timeout: 5–10 seconds
* retry: 2 or 3 attempts on `5xx`, `408`, network faults
* no retry on obvious client errors like invalid query params
* circuit breaker for repeated `5xx`/timeouts

With public SearXNG instances especially, you should expect inconsistent availability and disabled formats on some nodes, so retries should be conservative and errors should be explicit. ([SearXNG Documentation][1])

## Query builder

Centralize request construction so the rest of the code never manually concatenates strings.

```csharp
internal static class SearxngQueryBuilder
{
    public static string Build(SearchRequest request)
    {
        var query = HttpUtility.ParseQueryString(string.Empty);
        query["q"] = request.Query;
        query["format"] = "json";

        if (request.Page is int p) query["pageno"] = p.ToString(CultureInfo.InvariantCulture);
        if (!string.IsNullOrWhiteSpace(request.Language)) query["language"] = request.Language;
        if (!string.IsNullOrWhiteSpace(request.TimeRange)) query["time_range"] = request.TimeRange;
        if (request.SafeSearch is not null) query["safesearch"] = ((int)request.SafeSearch.Value).ToString();

        if (request.Categories?.Count > 0)
            query["categories"] = string.Join(",", request.Categories);

        if (request.Engines?.Count > 0)
            query["engines"] = string.Join(",", request.Engines);

        return query.ToString()!;
    }
}
```

Why centralize this:

* one place to encode provider parameter names
* one place to validate enums and lengths
* easier contract testing

SearXNG’s API docs and engine docs show search parameters such as `q`, `pageno`, `time_range`, `safe_search`/safe search semantics, and engine/category-style routing as part of its search system. ([SearXNG Documentation][1])

## Response parsing

Do **not** expose raw JSON directly as your public contract.

Instead:

* create internal DTOs that match SearXNG JSON
* map those DTOs into your stable public domain models

That protects your users if SearXNG adds fields or changes incidental payload structure.

Example:

```csharp
internal sealed class SearxngJsonResponse
{
    [JsonPropertyName("results")]
    public List<SearxngJsonResult>? Results { get; set; }

    [JsonPropertyName("number_of_results")]
    public long? NumberOfResults { get; set; }

    [JsonPropertyName("answers")]
    public List<string>? Answers { get; set; }

    [JsonPropertyName("corrections")]
    public List<string>? Corrections { get; set; }

    [JsonPropertyName("infoboxes")]
    public List<JsonElement>? Infoboxes { get; set; }
}

internal sealed class SearxngJsonResult
{
    [JsonPropertyName("title")]
    public string? Title { get; set; }

    [JsonPropertyName("url")]
    public string? Url { get; set; }

    [JsonPropertyName("content")]
    public string? Content { get; set; }

    [JsonPropertyName("engine")]
    public string? Engine { get; set; }
}
```

Map into:

```csharp
private static SearchResponse Map(
    SearxngJsonResponse dto,
    SearchRequest request,
    Uri baseUri,
    TimeSpan duration)
{
    var results = dto.Results?
        .Where(r => !string.IsNullOrWhiteSpace(r.Url))
        .Select(r => new SearchResult(
            r.Title ?? "",
            r.Url!,
            r.Content,
            r.Engine))
        .ToList() ?? [];

    return new SearchResponse(
        results,
        new SearchMetadata(
            baseUri.ToString(),
            request.Query,
            request.Page,
            duration,
            Partial: false,
            ResultCount: results.Count),
        []);
}
```

Use `System.Text.Json`, and configure it to be tolerant of extra fields.

## Error model

A production package should have **clear typed exceptions**:

```csharp
public abstract class SearxngException : Exception { ... }
public sealed class SearxngUnavailableException : SearxngException { ... }
public sealed class SearxngTimeoutException : SearxngException { ... }
public sealed class SearxngBadRequestException : SearxngException { ... }
public sealed class SearxngUnsupportedFormatException : SearxngException { ... }
public sealed class SearxngParseException : SearxngException { ... }
```

Suggested mapping:

* network failures / DNS / connect errors → `SearxngUnavailableException`
* timeout → `SearxngTimeoutException`
* 400-range with useful body → `SearxngBadRequestException`
* 403 when JSON is disabled on the instance → `SearxngUnsupportedFormatException`
* malformed/unexpected payload → `SearxngParseException`

That 403-on-disabled-format behavior is specifically documented by SearXNG. ([SearXNG Documentation][1])

## Instance strategy: self-hosted vs public

For production, I would strongly prefer one of these:

* **your own SearXNG instance**
* **a private trusted instance**
* **a list of failover instances you control**

I would not treat random public instances as dependable production infrastructure, because API format support can vary by instance configuration. The docs explicitly say supported response formats are defined in instance settings and may be disabled. ([SearXNG Documentation][1])

A good design is:

```csharp
public interface ISearxngEndpointSelector
{
    ValueTask<Uri> GetEndpointAsync(CancellationToken ct = default);
}
```

Then your package can support:

* single endpoint
* weighted pool
* health-checked failover

## Testing strategy

I’d use 4 kinds of tests.

### 1. Pure unit tests

For:

* query builder
* request validation
* DTO → domain mapping
* exception mapping

### 2. Contract tests

Feed saved JSON payloads into the parser:

* normal results
* empty results
* weird optional fields
* malformed JSON
* disabled format / 403 case

### 3. Integration tests

Run against:

* a local self-hosted SearXNG test instance
* or a controlled staging instance

* timeouts
* retries

### 4. Chaos/resilience tests

Simulate:

* slow upstream
* partial failures

## Example implementation sketch

```csharp
public sealed class SearxngClient : ISearxngClient
{
    private readonly HttpClient _httpClient;
    private readonly IOptions<SearxngOptions> _options;
    private readonly ILogger<SearxngClient> _logger;

    public SearxngClient(
        HttpClient httpClient,
        IOptions<SearxngOptions> options,
        ILogger<SearxngClient> logger)
    {
        _httpClient = httpClient;
        _options = options;
        _logger = logger;
    }

    public async Task<SearchResponse> SearchAsync(
        SearchRequest request,
        CancellationToken cancellationToken = default)
    {
        if (string.IsNullOrWhiteSpace(request.Query))
            throw new ArgumentException("Query is required.", nameof(request));

        var sw = Stopwatch.StartNew();
        var queryString = SearxngQueryBuilder.Build(request);
        var path = $"{_options.Value.SearchPath}?{queryString}";

        using var httpRequest = new HttpRequestMessage(HttpMethod.Get, path);
        using var response = await _httpClient.SendAsync(
            httpRequest,
            HttpCompletionOption.ResponseHeadersRead,
            cancellationToken);

        if (response.StatusCode == HttpStatusCode.Forbidden)
            throw new SearxngUnsupportedFormatException("JSON format may be disabled on this SearXNG instance.");

        if ((int)response.StatusCode >= 400 && (int)response.StatusCode < 500)
            throw new SearxngBadRequestException($"SearXNG returned {(int)response.StatusCode}.");

        response.EnsureSuccessStatusCode();

        await using var stream = await response.Content.ReadAsStreamAsync(cancellationToken);
        var dto = await JsonSerializer.DeserializeAsync<SearxngJsonResponse>(
            stream,
            JsonContext.Default.SearxngJsonResponse,
            cancellationToken);

        if (dto is null)
            throw new SearxngParseException("Response body was empty or invalid.");

        sw.Stop();

        var result = Map(dto, request, _httpClient.BaseAddress!, sw.Elapsed);

        _logger.LogInformation(
            "SearXNG search completed in {ElapsedMs} ms with {Count} results",
            sw.ElapsedMilliseconds,
            result.Results.Count);

        return result;
    }
}
```

## Design decisions I would make up front

### 1. Keep the public contract provider-neutral

Even if v1 only supports SearXNG, design as though you may later add Brave or another backend.

### 2. Prefer explicit options over magic defaults

Especially for:

* base URL
* timeout
* user agent
* retries
* default categories/engines

### 3. Fail clearly on capability mismatches

If JSON is disabled, say so directly.

### 4. Avoid leaking raw upstream payloads

Return clean models and preserve raw-body access only behind debugging hooks.

### 5. Make self-hosting the default production assumption

Public instances are better for experiments than hard production dependencies.

## The system around the package

If I were using this in a real app, I’d deploy it like this:

```text
App / API
  -> ISearxngClient
  -> typed HttpClient + resilience pipeline
  -> internal SearXNG instance
```

And around that:

* health checks for the configured SearXNG endpoint
* centralized logs/metrics
* query throttling if your app has bursty traffic
* caching for repeated identical queries if freshness requirements allow it

## My recommended v1 scope

Build only this first:

* `SearchAsync`
* request validation
* JSON parsing
* typed exceptions
* DI registration
* timeout + retry + circuit breaker
* structured logs
* unit + integration tests

Skip initially:

* autocomplete
* image/news specializations
* multi-instance routing
* caching inside the package
* raw generic query parameter bags

That gives you a clean, stable core.
